最近我在集中做一件事：尝试用 Google 开源的 `DESIGN.md` 协议，把一套产品的设计语言沉淀成 AI 可以读取、生成和校验的上下文。

一开始我对这件事的期待很直接：如果能把设计规范写成结构化文本，那么之后无论是做官网、产品界面，还是让 AI 辅助还原页面，都可以带着同一份设计上下文工作。它不只是给人看的规范，而是能进入 AI 工作流的设计输入。

但真正做下来，我发现难点不在“写一份 DESIGN.md”，而在“如何知道这份 DESIGN.md 真的有效”。一份文本规范看起来再完整，如果 AI follow 之后的结果跑偏，那它就还不能算可用。

这篇文章记录的是我这轮实践中逐渐摸出来的流程，以及几个非常具体的坑。

## DESIGN.md 不是完整设计系统，而是交换格式

在实践中，我越来越倾向于把 `DESIGN.md` 理解成一种设计上下文的交换格式。

它不像 Figma Library 或 Claude Design 里的 Design System 那样，是一个产品化的设计系统能力。后者通常包含可预览、可复用、可在工具内自动套用的组件和样式。

`DESIGN.md` 更像一个入口：它用 YAML 和 Markdown 把颜色、字体、间距、圆角、组件规则、品牌语气、Do / Don't、设计语言等信息表达出来，让 AI 工具能读懂并使用。它可以放进代码仓库，也可以作为 Claude Code、Codex、Stitch、Claude Design 等工具的上下文。

所以我后来不再追求让它“一次性记录完整设计系统”，而是把它拆成三层：

- Design Token：主要放在 YAML 中，记录颜色、字体、圆角、间距、阴影等可结构化调用的变量。
- Design Language：用 Markdown 描述产品应该呈现出的气质、层级、节奏、密度和视觉判断。
- Design Pattern：针对不同页面类型写具体范式，例如 Landing Page、Product UI、Document、Sign In。

这个拆分对后续迭代很关键。Token 决定基础材料，Design Language 决定方向，Design Pattern 决定具体场景里的落地方式。

## 第一步：先做一份可以被验证的 DESIGN.md

我的第一版做法，是先找调性接近的产品和公开 `DESIGN.md` 样本，从里面提取适合目标产品的视觉元素，再融合成一份初始 Dm。

这里我没有一上来就写很完整的组件库，而是先聚焦几个基础部分：

- 颜色：primary、secondary、tertiary、neutral、surface、on-surface、error、accent 等角色。
- 字体：headline、body、label 三类，再拆 display、lg、md、sm 等层级。
- 圆角、间距、基础按钮和卡片规则。
- 对整体气质的文字描述。

很快我就发现，components 不适合在早期写得过细。发散阶段如果把按钮、卡片、容器都定义得非常具体，AI 的探索空间会被锁死。因此我一度把 components 定义先移除，只保留备份，等风格方向收敛后再补回来。

这也是我这轮实践中的第一个判断：早期的 `DESIGN.md` 应该可验证，而不是追求完整。

## 坑 1：Stitch 直接导入 DESIGN.md 会曲解

我最初希望用 Stitch 来快速校验 Dm。方法很简单：把整理好的 `DESIGN.md` 粘贴到 Stitch 的 Start with your design 中，让它生成设计系统或页面，然后观察效果。

但结果并不理想。即使使用的是相对可信的公开样本，并且手动 review 过内容，Stitch 生成的设计系统仍然可能和原始设计语言差异很大。它会混入很多自身默认的设计倾向，甚至在 Dm 明确要求 light 风格时，仍然生成偏暗的视觉结果。

这个问题带来的麻烦是：我无法判断跑偏来自哪里。

是 `DESIGN.md` 写得不够清楚？是 token 不完整？还是 Stitch 自身模型和模板的偏置太强？

如果验证工具本身不稳定，规范就很难迭代。因为每一次结果不对，你都不知道该改规范，还是该换工具。

后来我对 Stitch 的定位做了调整：它不适合作为早期严格校验工具，更适合作为视觉发散工具。

## 坑 2：DESIGN.md 和 Bento Style Guide 之间不是透明互转

我也研究过 Google 公开仓库中与 `DESIGN.md` 相关的调用方式。能看到的是类似 `create_design_system_from_design_md` 这样的调用，但关键转换逻辑在 Stitch 服务器上，并没有开源。

这意味着，如果想搞清楚 `DESIGN.md` 如何变成 Bento Style Guide 或设计系统预览，就只能逆向猜测。

我当时的推测是，Stitch 并不是逐字段机械转换，而是大致经过这样的过程：

```text
DESIGN.md
  -> 解析 YAML frontmatter
  -> 解析 Markdown 正文中的设计语义
  -> 提取颜色、字体、字号、圆角与组件参数
  -> 生成或归一化内部 Theme
  -> 将 Theme 应用至固定 Bento Showcase 模板
```

也就是说，YAML token 只是一部分。Markdown 正文里的设计语义也会影响结果，而最终展示还会受到固定模板影响。

这解释了一个现象：你以为自己写的是“完整设计系统”，但工具可能只是从中抽取部分 token，再套入自己的 showcase 模板。结果看起来跑偏，并不一定是你写错了，而是中间转换规则并不透明。

这也让我意识到：`DESIGN.md` 的验证不能只依赖某一个工具的预览结果。

## 坑 3：一份 DESIGN.md 无法同时服务官网和产品 UI

早期我想过用同一份 Dm 同时产出官网设计和产品 UI。后来发现这条路很难走。

原因很简单：官网和产品 UI 的目标不同。

官网 Landing Page 需要叙事、品牌感、转化路径和视觉记忆点。它可以有更强的首屏设计、更明显的节奏变化、更具表达性的结构。

产品 UI 则更强调信息密度、可扫描性、操作效率、控件一致性和长时间使用的舒适度。它不能把官网里的视觉强调原样搬过来。

如果把两类场景都写进同一份 `DESIGN.md`，AI 很容易混用规则：在产品界面里使用过强的营销表达，或者在官网里套用过于工具化的后台结构。

我最后的判断是：

- Design Token 可以尽量共用。
- Design Language 可以共用一部分，但要保持抽象。
- Design Pattern 必须按场景拆分。

所以更合理的结构是：全局 Dm 记录基础 token 和设计语言，不同页面类型用独立 Pattern 补充。

## 坑 4：规范太详细，AI 反而无法发散

这也是非常实际的一坑。

当我把多个参考对象中的视觉细节融合到 Dm 里后，Claude Design 确实能比较稳定地 follow 它生成页面。但问题是，它太稳定了。稳定到即使设计语言之间有轻微矛盾，它也会尽量照单全收。

这在收敛阶段是优点，在发散阶段却是缺点。

如果还没有找到最终视觉方向，就提前把颜色、卡片、组件、布局语气写得太精确，AI 会被限制在一个很窄的空间里。生成结果看起来“符合规范”，但也可能像是多个参考产品的拼贴，缺少自己的调性。

后来我的做法是把发散和收敛拆开：

- 发散时，删减 Dm，只保留核心气质和关键视觉线索，甚至移除部分颜色。
- 让 Stitch 每轮生成多个方向，挑一个继续 loop。
- 每轮根据看到的结果，反向调整 Dm，把真正有价值的元素沉淀下来。
- 收敛时，再把 Dm 导回 Claude Design 生成页面并细修。

这一步的经验是：`DESIGN.md` 不是越详细越好。它应该根据阶段调节颗粒度。

## Claude Design 更适合收敛和精修

在试过 Stitch 校验和 Figma MCP 绘制之后，我开始转向 Claude Design。

Figma MCP + Codex 的方式理论上很直接：让 AI 在 Figma 里按 Dm 绘制。但实际问题是速度慢，而且在 Figma 中做的微调很难快速同步回 Dm。它可以作为某些场景下的实现方式，但不适合高频探索。

Claude Design 本质上也是生成 HTML，但它有几个更适合设计迭代的能力：

- 可以直接基于 Dm 生成初始页面。
- 可以通过 comments、mark up、edit 等方式逐步修改。
- 可以用 Tweaks 在局部设计元素上快速尝试变体。
- 生成结果更容易进入后续代码和部署流程。

不过我也发现，Tweaks 不能替代 Stitch 的发散。Tweaks 更像局部变量调整，例如颜色、圆角、透明度、某些控件形态；Stitch 更擅长大胆探索布局结构和整体视觉方向。

所以我现在会这样分工：

- Stitch：用于方向发散。
- Claude Design：用于基于方向做页面收敛和细节修改。
- Claude Code / Codex：用于更严格地 follow Dm 生成 HTML，测试 Dm 本身是否健壮。

## 坑 5：单个 HTML 提取 Design System 会过拟合

当首页设计基本确定后，我尝试把 Claude Design 生成并打磨过的首页 HTML 上传，让它提取 Design System，再基于这个 DS 生成一个二级页面。

结果不太理想：二级页面太像首页。

这个问题其实很合理。首页里有很多只属于首页的设计，比如 Hero 的强表达、关键词强调、轻微动效、首屏视觉节奏。它们对首页是优点，但如果被提取成全局设计系统规则，就会污染其他页面。

单个 HTML 让 AI 分不清：

- 哪些是全局 token。
- 哪些是可复用组件。
- 哪些是首页专属表达。
- 哪些只是某一轮修改留下来的局部方案。

所以，只用单页 HTML 提取 Design System 很容易过拟合。

更合理的做法应该是：至少提供多个页面或多个状态，让 AI 看到设计语言如何跨场景复用。比如首页、价格页、登录页、产品页、文档页。只有这样，AI 才更容易判断什么是系统规则，什么是页面特例。

## 我后来采用的三层结构

经过几轮调整后，我把 Dm 拆成了三层：

第一层是 Design Token。

它主要是 YAML，记录颜色、字体、圆角、间距、阴影、transition 等结构化变量。但我不再把 token 当作努力的终点，因为颜色和数值随时可以替换。

第二层是 Design Language。

这一层用来描述产品真正的设计调性：界面应该克制还是张扬，信息密度应该高还是低，圆角如何体现，按钮和容器的语义关系是什么，动效应该服务什么，哪些常见视觉套路要避免。

在 review 时，我还发现 Design Language 里最好不要写太多具体数值。比如不要写死某个颜色值或某个具体字体名，而是描述它的角色、气质和使用原则。这样后续 token 变化时，语言层不需要频繁重写。

第三层是 Design Pattern。

这一层针对具体场景。比如 Landing Page、Product UI、Document、Sign In 各自写不同规则。像 Cards & Containers、Composition 这类容易和具体页面绑定的内容，就更适合放进 Pattern，而不是放在全局 Design Language 里。

我的当前判断是：全局 Dm 不应该承载所有页面细节。它应该提供设计语言和基础材料，具体页面由 Pattern 接力。

## 校验时反而要用“更笨”的工具

一个很有意思的发现是：Claude Design 生成效果好，并不代表 Dm 本身足够健壮。

原因是 Claude Design 可能有自己的内置工作流，会帮你补齐、修正或消解 Dm 中的模糊点。结果看起来不错，但你无法知道到底是 Dm 写得好，还是工具帮你兜住了。

所以后续校验时，我反而倾向于使用 Claude Code、Codex 这类更直接的代码工具，让它们 follow Dm 生成 HTML。

这些工具有时更“笨”，但正因为如此，它们更容易暴露 Dm 的问题：

- 圆角规则没写清楚，Sticky Toolbar 就可能过圆。
- Button 类型没拆清楚，fill、outline、ghost、destructive 就会混用。
- Cards & Containers 如果写在全局语言里，其他页面也会误用 Landing Page 的卡片。
- Do / Don't 如果重复或太抽象，模型就会忽略。

聪明工具适合出效果，直接工具适合做压力测试。这个分工对我很有启发。

## 反 AI 味也应该成为规范的一部分

这轮实践里我越来越明确一个问题：AI 生成物的问题通常不是“不可用”，而是“太像 AI”。

它会自然倾向于一些常见套路：大渐变、泛用卡片、过度圆角、装饰性图标、空泛指标、漂亮但无意义的布局。这些东西满足基本可用性，所以模型会认为是好设计。但人一看就知道它缺少具体判断。

因此，`DESIGN.md` 或相关 Skill 里需要加入反 AI 味规则：

- 哪些常见视觉套路不要用。
- 哪些组件不要滥用卡片和阴影。
- 缺少真实数据时不要编造数字。
- 不要把装饰当作信息层级。
- 不要让 Landing Page 的表达性污染 Product UI。
- 不要把局部页面特例提炼成全局规则。

这部分不一定全部写进 `DESIGN.md`，也可以放在 Skill 或 Checklist 里。但它必须出现在 AI 工作流的上下文中。

## 当前我认可的工作流

如果只总结这轮之后我会继续沿用的流程，大概是这样：

1. 找到调性接近的参考对象和公开 Dm 样本。
2. 写一版轻量 Dm，只包含核心 token、设计语言和少量禁忌。
3. 用 Claude Design 生成 HTML，验证 Dm 是否能产出可接受的初始方向。
4. 如果方向不够有生命力，把 Dm 删减后交给 Stitch 发散。
5. 从 Stitch 的多个方向中选取有效元素，反向更新 Dm。
6. 再导回 Claude Design 做页面收敛、评论修改和局部变体尝试。
7. 用更直接的代码工具 follow Dm 生成页面，暴露规范缺口。
8. 把全局 Design Language 和具体 Design Pattern 拆开维护。
9. 用多个页面测试泛化能力，而不是只看首页。
10. 持续补充反 AI 味规则和更细的 token 使用原则。

这套流程的核心不是一次生成完美页面，而是形成循环：生成、观察、修正、再生成。

## 最后：DESIGN.md 的价值在于可迭代

这次实践让我对 `DESIGN.md` 的理解更务实了。

它不是魔法文件，也不是把设计系统文本化之后就能自动解决一切。它真正有价值的地方，是把设计判断变成 AI 可以消费的上下文，并且能在生成结果中反复被检验。

好的 Dm 不应该只是一份“看起来很完整”的文档。它应该能回答：

- AI 生成时应该遵循什么气质？
- 哪些规则是全局的，哪些只是某类页面的？
- 当工具生成跑偏时，我应该改 token、改语言，还是改 pattern？
- 生成物出现 AI 味时，规范里是否有足够明确的反例和禁忌？
- 换一个工具 follow 它，结果是否仍然稳定？

我现在更愿意把 `DESIGN.md` 看作一份活文档。它需要和页面生成、视觉 review、工具校验、Pattern 拆分一起迭代。只有进入这个循环，它才不只是“设计规范的 Markdown 版本”，而是真正能服务 AI 设计工作流的上下文协议。
